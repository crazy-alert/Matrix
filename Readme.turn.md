# Настройка TURN-сервера (Coturn) для Matrix

TURN (Traversal Using Relays around NAT) сервер необходим для организации аудио- и видеозвонков в Matrix, когда два клиента не могут установить прямое peer-to-peer соединение (например, из-за симметричного NAT или корпоративного файрвола). В данном стеке используется Coturn — свободная реализация TURN/STUN сервера.

Файл конфигурации `turnserver.conf` задаёт все параметры работы TURN-сервера. В Docker-образе `instrumentisto/coturn` ожидается, что этот файл будет смонтирован в контейнер. В нашем случае используется шаблон `turnserver.conf.template`, в котором переменные заменяются через `envsubst` на лету.

## Основные настройки turnserver.conf

Ниже описаны ключевые параметры, необходимые для корректной работы TURN в связке с Matrix (Synapse).

### Общие параметры

| Параметр | Описание | Пример значения |
|----------|----------|-----------------|
| `listening-port` | Порт для UDP и TCP (если включено) прослушивания TURN. Обычно 3478. | `listening-port=3478` |
| `tls-listening-port` | Порт для TLS (DTLS) прослушивания. Обычно 5349. | `tls-listening-port=5349` |
| `fingerprint` | Добавлять атрибут FINGERPRINT в сообщения TURN (требуется для совместимости). | `fingerprint` |
| `lt-cred-mech` | Включить долговременный механизм аутентификации (необходим для Matrix). | `lt-cred-mech` |
| `use-auth-secret` | Включить аутентификацию на основе общего секрета (shared secret). В паре с `static-auth-secret`. | `use-auth-secret` |
| `static-auth-secret` | Секретная строка, которая используется для генерации временных паролей (credentials). Должна совпадать с `turn_shared_secret` в конфигурации Synapse. | `static-auth-secret=ваш-секрет` |
| `realm` | Доменное имя, которое TURN-сервер будет сообщать клиентам. Обычно совпадает с основным доменом Matrix (например, `msk.kleomars.top`). | `realm=msk.kleomars.top` |

### Параметры сети

| Параметр | Описание | Пример значения |
|----------|----------|-----------------|
| `external-ip` | Внешний IP-адрес сервера (если сервер находится за NAT). Можно указать несколько через запятую, если есть несколько внешних адресов. | `external-ip=109.107.189.155` |
| `relay-ip` | Внутренний IP-адрес, на котором TURN будет ретранслировать трафик. Обычно совпадает с внешним или указывается адрес интерфейса. | `relay-ip=109.107.189.155` |
| `min-port` и `max-port` | Диапазон портов, которые TURN будет использовать для ретрансляции медиатрафика. Должен быть открыт в фаерволе (UDP). | `min-port=49160`<br>`max-port=49200` |
| `no-tcp-relay` | Запретить ретрансляцию TCP (если не нужна). Для звонков обычно достаточно UDP. | `no-tcp-relay` |
| `no-udp-relay` | Запретить ретрансляцию UDP (не включайте, если нужны звонки). | (не указывать) |
| `no-tls` и `no-dtls` | Отключить TLS/DTLS, если не планируется использовать защищённые соединения. Обычно оставляют включённым. | (не указывать, чтобы TLS работал) |

### Безопасность и ограничения

| Параметр | Описание | Пример значения |
|----------|----------|-----------------|
| `total-quota` | Максимальное количество одновременно обслуживаемых клиентов. | `total-quota=100` |
| `bps-capacity` | Максимальная пропускная способность (бит/с). | `bps-capacity=0` (без ограничений) |
| `stale-nonce` | Время жизни nonce (в секундах) для аутентификации. | `stale-nonce=600` |
| `no-multicast-peers` | Запретить мультикаст-адреса в качестве пиров. | `no-multicast-peers` |
| `no-stun` | Отключить STUN-функциональность (если нужен только TURN). | (не указывать, STUN полезен) |

### Логирование

| Параметр | Описание | Пример значения |
|----------|----------|-----------------|
| `Verbose` | Подробное логирование (полезно при отладке). | `Verbose` |
| `log-file` | Путь к файлу лога. В контейнере обычно перенаправляется в stdout/stderr. | (не указывать, если логи идут в консоль) |

## Пример готового turnserver.conf (для использования с Matrix)

```conf
# Слушаем на стандартных портах
listening-port=3478
tls-listening-port=5349

# Включаем необходимые механизмы
fingerprint
lt-cred-mech
use-auth-secret
static-auth-secret=${COTURN_STATIC_AUTH_SECRET}

# Realm – должен совпадать с доменом вашего сервера Matrix
realm=${DOMAIN}

# Внешний и внутренний IP
external-ip=${COTURN_EXTERNAL_IP}
relay-ip=${COTURN_INTERNAL_IP}

# Диапазон портов для ретрансляции
min-port=49160
max-port=49200

# Ограничения
total-quota=100
bps-capacity=0
stale-nonce=600

# Безопасность
no-multicast-peers
no-tcp-relay

# Логирование (для отладки можно включить verbose)
Verbose
```
### Переменные окружения, используемые в шаблоне
В нашем стеке turnserver.conf.template содержит плейсхолдеры вида ${VAR}, которые заменяются через envsubst при запуске контейнера. Вот какие переменные должны быть определены в .env:

- `COTURN_STATIC_AUTH_SECRET` — общий секрет для аутентификации (генерируется автоматически или задаётся вручную).

- `COTURN_EXTERNAL_IP` — внешний IP сервера.

- `COTURN_INTERNAL_IP` — внутренний IP (часто совпадает с внешним).

- `DOMAIN` — основной домен (используется как realm).

 ### Важные замечания по настройке
Совпадение секрета — значение static-auth-secret должно быть точно таким же, как turn_shared_secret в конфигурации Synapse (homeserver.yaml). Именно через этот секрет Synapse выдаёт временные credentials клиентам для использования TURN.

#### Открытые порты — обязательно откройте в фаерволе:

 - 3478/tcp и 3478/udp — для TURN (TCP может использоваться для fallback).

 - 5349/tcp и 5349/udp — для TURN over TLS/DTLS (если используете).

 - Диапазон ретрансляции (например, 49160-49200/udp).

`Realm` — должен совпадать с доменом, указанным в server_name Synapse (или с основным доменом). Некоторые клиенты могут проверять это соответствие.

________
Проверка работы TURN — после запуска можно проверить, отвечает ли сервер на STUN-запросы, например, с помощью онлайн-инструментов или утилиты stunclient.

### Типичные ошибки
 - Не совпадает `static-auth-secret` — клиенты не могут получить разрешение на использование TURN.

- Неправильно указан `external-ip` — клиенты получают адрес, который не маршрутизируется.

- Закрыты порты в фаерволе — соединения не проходят.

- Не указан `realm` — некоторые клиенты могут отвергать подключение.

# Типичные проблемы и их решение
## ❌ Клиент не получает TURN-кандидаты
 - Проверьте, что в `homeserver.yaml` правильно указаны `turn_uris` и `turn_shared_secret`.

- Убедитесь, что секрет одинаков на `Synapse` и `Coturn`.

- Проверьте, что порты TURN открыты (3478/udp, 5349/tcp, диапазон ретрансляции).

- Проверьте логи Synapse на наличие ошибок при запросе /turn.

## ❌ TURN-сервер не отвечает
- Проверьте, запущен ли контейнер coturn: `docker ps | grep coturn`.

- Посмотрите логи: `docker logs coturn`.

- Проверьте, слушает ли процесс порты внутри контейнера: `docker exec coturn netstat -tulpn`.

- Убедитесь, что `external-ip` указан верно и сервер видит правильный внешний адрес.

## ❌ Ошибки аутентификации на TURN (401 Unauthorized)
- Секреты не совпадают.

- Время жизни `credentials` истекло (попробуйте перезапустить клиент).

- Используется неправильный транспорт (TCP вместо UDP и наоборот).

## ❌ Звонки работают, но только после долгого ожидания
- Возможно, клиент перебирает много ICE-кандидатов, включая нерабочие. Проверьте, правильно ли настроены STUN/TURN урлы.

- Диапазон портов для ретрансляции слишком мал или занят другими процессами.

## ❌ TURN работает, но только для одного клиента
- Проверьте `total-quota` и `bps-capacity`, возможно, они слишком малы.
